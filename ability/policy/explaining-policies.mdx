title: How Vincent Policies Work
---

Vincent Policies are programmable guardrails that determine whether Vincent Apps can execute specific Abilities on behalf of users. They ensure autonomous agents operate strictly within user-defined boundaries.

## What is a Vincent Policy?

A Vincent Policy is a function built using Lit Actions that provides user-configurable parameters and sophisticated validation logic for Ability executions.

## Real-World Policy Examples

<Tabs>
  <Tab title="Financial Controls">
    - **Daily Spending Limits**: Track cumulative spending and deny executions when exceeded
    - **Multi-Signature Requirements**: Require additional approvals for high-value transactions
    - **Token Allowlists**: Restrict transactions to specific tokens or verified contracts
  </Tab>
  <Tab title="Access Management">
    - **Membership Gates**: Verify ownership of assets or roles before allowing access
    - **Time-Based Restrictions**: Allow execution only during specific hours or days
    - **Usage Limits**: Implement rate limiting and cooldown periods
  </Tab>
  <Tab title="Risk Management">
    - **Pattern Analysis**: Monitor spending patterns and flag suspicious activity
    - **Circuit Breakers**: Automatically disable abilities when unusual activity detected
    - **Emergency Stops**: Admin-controlled pauses for policy-governed operations
  </Tab>
</Tabs>

## Policy Lifecycle

Vincent Policies execute in three phases to provide comprehensive validation and state management:

<Steps>
  <Step title="Precheck">
    Local validation to check if policy should permit execution based on current conditions
  </Step>

  <Step title="Evaluate">
    In-action validation with full blockchain access to make final allow/deny decision
  </Step>

  <Step title="Commit (Optional)">
    Update policy state after successful Ability execution to maintain accurate tracking
  </Step>
</Steps>

## Defining Your Policy

<ParamField path="packageName" type="string" required>
  The npm package name of your Policy (e.g., `@your-org/policy-name`)
</ParamField>

<ParamField path="abilityParamsSchema" type="ZodSchema" required>
  Zod schema defining what parameters Abilities must provide to your Policy
</ParamField>

<ParamField path="userParamsSchema" type="ZodSchema" required>
  Zod schema for user-configurable boundaries and limits stored on-chain
</ParamField>

<ParamField path="precheckAllowResultSchema" type="ZodSchema">
  Zod schema for successful precheck return values
</ParamField>

<ParamField path="precheckDenyResultSchema" type="ZodSchema">
  Zod schema for denied precheck return values
</ParamField>

<ParamField path="precheck" type="function" required>
  Async function providing fast, local validation before using network resources
  <Expandable title="function signature">
    ```typescript
    (params: { abilityParams, userParams }, context: PolicyContext) => Promise<PrecheckResult>
    ```
  </Expandable>
</ParamField>

<ParamField path="evalAllowResultSchema" type="ZodSchema">
  Zod schema for successful evaluation return values
</ParamField>

<ParamField path="evalDenyResultSchema" type="ZodSchema">
  Zod schema for denied evaluation return values
</ParamField>

<ParamField path="evaluate" type="function" required>
  Async function with full blockchain access making final allow/deny decision
  <Expandable title="function signature">
    ```typescript
    (params: { abilityParams, userParams }, context: PolicyContext) => Promise<EvaluateResult>
    ```
  </Expandable>
</ParamField>

<ParamField path="commitParamsSchema" type="ZodSchema">
  Zod schema for parameters passed to commit function
</ParamField>

<ParamField path="commitAllowResultSchema" type="ZodSchema">
  Zod schema for successful commit return values
</ParamField>

<ParamField path="commitDenyResultSchema" type="ZodSchema">
  Zod schema for failed commit return values
</ParamField>

<ParamField path="commit" type="function">
  Optional async function to update policy state after successful Ability execution
  <Expandable title="function signature">
    ```typescript
    (params: CommitParams, context: PolicyContext) => Promise<CommitResult>
    ```
  </Expandable>
</ParamField>

## Policy Context

The `policyContext` object provides essential information and utilities:

<ParamField path="abilityIpfsCid" type="string">
  IPFS CID of the executing Ability
</ParamField>

<ParamField path="appId" type="number">
  ID of the Vincent App executing this Ability
</ParamField>

<ParamField path="appVersion" type="number">
  Version of the Vincent App
</ParamField>

<ParamField path="delegation" type="object">
  Information about the execution delegation
  <Expandable title="properties">
    <ParamField path="delegateeAddress" type="string">
      Ethereum address of the Ability executor
    </ParamField>
    <ParamField path="delegatorPkpInfo" type="object">
      User's Vincent Wallet information (tokenId, ethAddress, publicKey)
    </ParamField>
  </Expandable>
</ParamField>

<ParamField path="allow" type="function">
  Helper to return allow results
</ParamField>

<ParamField path="deny" type="function">
  Helper to return deny results
</ParamField>

## Creating Vincent Policies

Vincent Policies are created using `createVincentPolicy` from the Vincent Ability SDK:

```typescript
import { createVincentPolicy } from '@lit-protocol/vincent-ability-sdk';

export const vincentPolicy = createVincentPolicy({
  packageName: '@your-org/my-policy',

  abilityParamsSchema,
  userParamsSchema,

  precheckAllowResultSchema,
  precheckDenyResultSchema,
  precheck: async ({ abilityParams, userParams }, policyContext) => {
    // Local validation logic
  },

  evalAllowResultSchema,
  evalDenyResultSchema,
  evaluate: async ({ abilityParams, userParams }, policyContext) => {
    // Final validation with blockchain access
  },

  commitParamsSchema,
  commitAllowResultSchema,
  commitDenyResultSchema,
  commit: async (params, policyContext) => {
    // Update policy state after successful execution
  },
});
```
