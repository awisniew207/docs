Vincent Policies are programmable guardrails that determine whether Vincent Apps can execute specific Abilities on behalf of users. They ensure autonomous agents operate strictly within user-defined boundaries.

## What is a Vincent Policy?

A Vincent Policy is a function built using Lit Actions that provides user-configurable parameters and sophisticated validation logic for Ability executions.

## Policy Lifecycle

Vincent Policies execute in three phases to provide comprehensive validation and state management:

<Steps>
  <Step title="Precheck">
    Local validation to check if policy should permit execution based on current conditions
  </Step>

  <Step title="Evaluate">
    In-action validation with full blockchain access to make final allow/deny decision
  </Step>

  <Step title="Commit (Optional)">
    Update policy state after successful Ability execution to maintain accurate tracking
  </Step>
</Steps>

## Defining Your Policy

Vincent Policies are created using `createVincentPolicy` from the Vincent Ability SDK:

```typescript
import { createVincentPolicy } from '@lit-protocol/vincent-ability-sdk';

export const vincentPolicy = createVincentPolicy({
  packageName: '@your-org/my-policy',

  // Parameter schemas
  abilityParamsSchema,    // What Abilities pass to your Policy
  userParamsSchema,       // User-configurable boundaries and limits

  // Precheck validation
  precheckAllowResultSchema,
  precheckDenyResultSchema,
  precheck: async ({ abilityParams, userParams }, policyContext) => {
    // Local validation logic
  },

  // In-action evaluation
  evalAllowResultSchema,
  evalDenyResultSchema,
  evaluate: async ({ abilityParams, userParams }, policyContext) => {
    // Final validation with blockchain access
  },

  // State management
  commitParamsSchema,
  commitAllowResultSchema,
  commitDenyResultSchema,
  commit: async (params, policyContext) => {
    // Update policy state after successful execution
  },
});
```

## Parameter Schemas

### abilityParamsSchema

Defines what parameters Abilities must provide to your Policy:

```typescript
import { z } from 'zod';

const abilityParamsSchema = z.object({
  tokenAddress: z.string(),
  amount: z.number(),
});
```

### userParamsSchema

Defines user-configurable boundaries stored on-chain:

```typescript
const userParamsSchema = z.object({
  dailySpendingLimit: z.number(),
  allowedTokens: z.array(z.string()).optional(),
});
```

## Precheck Function

Provides fast, local validation before using network resources:

```typescript
precheck: async ({ abilityParams, userParams }, policyContext) => {
  const { amount, tokenAddress } = abilityParams;
  const { dailySpendingLimit, allowedTokens } = userParams;

  // Check token allowlist
  if (allowedTokens && !allowedTokens.includes(tokenAddress)) {
    return policyContext.deny({
      reason: 'Token not allowed',
      allowedTokens,
    });
  }

  // Check spending limit
  const currentSpending = await getCurrentDailySpending(tokenAddress);
  if (currentSpending + amount > dailySpendingLimit) {
    return policyContext.deny({
      reason: 'Spending limit exceeded',
      currentSpending,
      dailySpendingLimit,
    });
  }

  return policyContext.allow({
    currentSpending,
    dailySpendingLimit,
  });
}
```

## Evaluate Function

Final validation with full blockchain access in the Lit Action environment:

```typescript
evaluate: async ({ abilityParams, userParams }, policyContext) => {
  // Similar logic to precheck but with blockchain access
  // This is the final decision point for allow/deny

  const validation = await performBlockchainValidation();

  if (validation.success) {
    return policyContext.allow(validation.data);
  } else {
    return policyContext.deny({ reason: validation.error });
  }
}
```

## Commit Function

Updates policy state after successful Ability execution:

```typescript
commit: async (params, policyContext) => {
  const { spentAmount, tokenAddress } = params;

  try {
    const result = await updateSpendingRecord({
      appId: policyContext.appId,
      userAddress: policyContext.delegation.delegatorPkpInfo.ethAddress,
      spentAmount,
      tokenAddress,
    });

    return policyContext.allow({
      updatedDailySpending: result.updatedSpending,
      remainingLimit: result.remainingLimit,
    });
  } catch (error) {
    return policyContext.deny({
      reason: 'Failed to update spending record',
      error: error.message,
    });
  }
}
```

<Warning>
Always implement commit functions for stateful Policies. Failure to update state can lead to incorrect policy decisions in future executions.
</Warning>
